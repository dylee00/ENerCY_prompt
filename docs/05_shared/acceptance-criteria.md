# Acceptance Criteria

## 문서 목적

이 문서는 AI 기반 에너지 조달 의사결정 지원 솔루션의 완료 기준을 정의한다.

이 문서의 목적은 다음과 같다.

- 구현 완료 여부를 일관된 기준으로 판단한다.
- 데이터, 모델, 백엔드, 프론트엔드의 결과물을 같은 기준에서 검증한다.
- 문서와 구현 간 불일치를 줄인다.
- 병렬 작업 시 팀 간 기대치를 맞춘다.

이 문서는 기능 구현 여부뿐 아니라, 문서화, 설명 가능성, 재현 가능성, 도메인 간 정합성까지 포함한다.

---

## 공통 완료 조건

아래 조건은 모든 주요 작업에 공통으로 적용된다.

- 관련 문서가 최신 상태로 갱신되어 있어야 한다.
- 구현 내용이 문서와 일치해야 한다.
- 다른 도메인에 영향을 주는 변경은 계약 문서에 반영되어 있어야 한다.
- 숨겨진 가정이나 매직 넘버가 없어야 한다.
- 다음 작업자가 문맥 없이도 이어서 작업할 수 있어야 한다.

---

## 기능 요구사항 완료 기준

### 1. 유가 예측 기능

완료 기준:
- 시스템이 유가 시계열 데이터를 기반으로 미래 구간을 예측할 수 있어야 한다.
- 예측 모델은 LSTM 기반이어야 한다.
- 예측 결과는 현재 시점 기준 미래 예측 시점들의 값으로 제공되어야 한다.
- 예측 결과를 바탕으로 미래 구간 내 최대 예측 유가와 최소 예측 유가를 계산할 수 있어야 한다.

검증 질문:
- 예측 모델이 실제로 LSTM 기반인가?
- 모델 출력에서 미래 시점별 예측값을 확인할 수 있는가?
- 최대값, 최소값을 계산할 수 있는 구조인가?

---

### 2. 공통 하이퍼파라미터 정책

완료 기준:
- 오일 종류와 무관하게 동일한 하이퍼파라미터 정책을 사용해야 한다.
- 하이퍼파라미터는 문서화되어 있어야 한다.
- 하이퍼파라미터 값은 코드와 문서에서 일치해야 한다.

검증 질문:
- 오일 종류별로 다른 하이퍼파라미터 세트를 사용하지 않는가?
- 하이퍼파라미터 정책이 별도 문서에 기록되어 있는가?
- 실험 로그 또는 설정 파일에서 동일 정책을 확인할 수 있는가?

---

### 3. MLflow 기반 실험 추적 및 최적화

완료 기준:
- 모델 학습 실험 결과가 MLflow에 기록되어야 한다.
- 주요 실험 정보가 추적 가능해야 한다.
- 어떤 모델이 선택되었는지 근거를 확인할 수 있어야 한다.

최소 기록 항목:
- 실험 이름
- 모델 종류
- 하이퍼파라미터
- 데이터 분할 기준
- 평가 지표
- 실행 시각 또는 실행 식별 정보

검증 질문:
- MLflow에 실험 로그가 남는가?
- 어떤 조건으로 모델이 선택되었는지 확인 가능한가?

---

### 4. 성능 모니터링 및 재학습 트리거

완료 기준:
- 성능 모니터링 정책이 문서화되어 있어야 한다.
- 재학습 트리거 조건이 명시되어 있어야 한다.
- 성능 저하 또는 기준 이탈 시 재학습 여부를 판단할 수 있어야 한다.

검증 질문:
- 모니터링 기준 지표가 문서로 정리되어 있는가?
- 어떤 조건에서 재학습을 트리거하는지 명확한가?
- 코드나 설계 상에서 해당 정책을 구현 가능한가?

---

### 5. 구매 추천 기능

완료 기준:
- 시스템이 threshold 정책에 따라 현재 시점의 구매 추천 결과를 제공해야 한다.
- 구매 추천은 단순 출력이 아니라 문서화된 규칙에 기반해야 한다.
- 추천 결과는 설명 가능한 근거와 연결되어야 한다.

최소 출력 요소:
- 현재 판단 시점
- 미래 예측 구간 내 최대 예측 유가
- 미래 예측 구간 내 최소 예측 유가
- 추천 결과
- 추천 판단 기준 또는 요약 설명

검증 질문:
- 추천 결과가 threshold 정책과 연결되어 있는가?
- 추천 결과를 백엔드 또는 UI에서 설명할 수 있는가?

---

## 데이터 도메인 완료 기준

완료 기준:
- 데이터셋 출처가 문서화되어 있어야 한다.
- 주요 컬럼과 스키마가 정의되어 있어야 한다.
- 전처리 정책이 문서화되어 있어야 한다.
- train/validation/test split 기준이 정리되어 있어야 한다.
- 오일 종류 정의 기준이 문서화되어 있어야 한다.
- 시계열 정렬 기준과 윈도우 생성 기준이 문서화되어 있어야 한다.

검증 질문:
- 입력 데이터가 어떤 구조인지 문서만 보고 이해 가능한가?
- 결측치, 이상치, 정렬, 분할 정책이 명확한가?
- 모델 입력으로 어떻게 변환되는지 추적 가능한가?

---

## 모델 도메인 완료 기준

완료 기준:
- LSTM 구조가 문서화되어 있어야 한다.
- 입력 시퀀스 길이와 예측 horizon이 정의되어 있어야 한다.
- 출력 형식이 문서화되어 있어야 한다.
- 평가지표가 정의되어 있어야 한다.
- 추론 출력이 백엔드에서 활용 가능한 형태여야 한다.

검증 질문:
- 모델 입력과 출력이 명확한가?
- 최대/최소 예측 유가를 계산할 수 있는 출력 구조인가?
- 평가 기준이 모델 선택과 연결되는가?

---

## 백엔드 도메인 완료 기준

완료 기준:
- API 계약 문서가 존재해야 한다.
- 예측 결과 응답 구조가 정의되어 있어야 한다.
- 추천 결과 응답 구조가 정의되어 있어야 한다.
- 에러 응답 규칙이 정의되어 있어야 한다.
- 프론트엔드가 필요한 정보를 백엔드 응답만으로 받을 수 있어야 한다.

최소 API 요구사항:
- 예측 조회 API
- 구매 추천 조회 API
- 모니터링 상태 조회 API 또는 이에 준하는 기능

검증 질문:
- 프론트엔드가 필요한 데이터를 API 계약만 보고 사용할 수 있는가?
- 추천 로직이 프론트엔드가 아니라 백엔드 기준으로 정리되어 있는가?

---

## 프론트엔드 도메인 완료 기준

완료 기준:
- 사용자가 현재 판단 시점의 추천 맥락을 이해할 수 있어야 한다.
- 예측 결과가 시각적으로 확인 가능해야 한다.
- 최대/최소 예측 유가가 명확하게 표시되어야 한다.
- 구매 추천 결과가 사용자 친화적으로 표현되어야 한다.
- 모니터링 상태 또는 모델 상태가 필요한 수준에서 표시되어야 한다.

최소 UI 요구사항:
- 대시보드 또는 메인 화면
- 예측 시각화 요소
- 최대/최소 유가 표시 카드
- 구매 추천 카드
- 상태 표시 영역

검증 질문:
- 사용자가 한 화면 흐름에서 판단 근거를 이해할 수 있는가?
- 프론트엔드가 비즈니스 로직을 재구현하지 않고 계약된 응답만 소비하는가?

---

## 통합 완료 기준

완료 기준:
- 데이터 → 모델 → 백엔드 → 프론트엔드 흐름이 연결되어 있어야 한다.
- 모델 출력과 백엔드 응답 계약이 일치해야 한다.
- 백엔드 응답과 프론트엔드 UI 요구사항이 일치해야 한다.
- threshold 정책이 문서, 백엔드, UI 설명에서 서로 모순되지 않아야 한다.

검증 질문:
- end-to-end 흐름을 설명할 수 있는가?
- 특정 추천 결과가 어떤 예측값과 어떤 정책에서 나왔는지 추적 가능한가?

---

## 문서화 완료 기준

완료 기준:
- 아래 문서가 존재하거나 최소 초안이 있어야 한다.
  - 프로젝트 개요 문서
  - 데이터 스키마 문서
  - 전처리 정책 문서
  - LSTM 모델링 명세
  - 하이퍼파라미터 정책 문서
  - MLflow 추적 정책 문서
  - threshold 정책 문서
  - 백엔드 API 계약 문서
  - 프론트엔드 UI 명세 문서
  - assumptions 문서
  - decision log 문서

검증 질문:
- 작업자가 구현 전에 문서만 읽고 전체 구조를 이해할 수 있는가?
- 중요한 설계 변경 사항이 decision log에 남아 있는가?

---

## 완료 정의 체크리스트

아래 체크리스트를 모두 만족하면 해당 범위의 작업을 완료로 본다.

- [ ] 관련 문서가 작성 또는 갱신되었다.
- [ ] 구현이 문서와 일치한다.
- [ ] LSTM 기반 모델 요구사항을 만족한다.
- [ ] 오일 종류와 무관한 공통 하이퍼파라미터 정책을 유지한다.
- [ ] MLflow 기반 실험 추적이 가능하다.
- [ ] 성능 모니터링 및 재학습 기준이 문서화되어 있다.
- [ ] threshold 기반 구매 추천 로직이 정의되어 있다.
- [ ] 최대/최소 미래 예측 유가를 제시할 수 있다.
- [ ] 백엔드 API 계약이 프론트엔드 요구사항을 만족한다.
- [ ] 프론트엔드에서 추천 결과를 이해 가능한 형태로 표시한다.
- [ ] assumptions와 decision log가 최신 상태다.
- [ ] 다음 작업자가 그대로 이어서 작업할 수 있다.
