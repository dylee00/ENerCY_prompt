## 지금 가장 먼저 해야 할 일

### 1. 역할별 프롬프트 통합 검토

검토 대상:
- 데이터 입력 구조
- 모델 출력 구조
- recommendation 응답 구조
- 프론트엔드 필요 필드

- monitoring 범위

완료 기준:
- 같은 개념을 도메인마다 다르게 정의하지 않는다.
- 병렬 구현 전에 공통 계약을 최소 수준으로 맞춘다.

---

### 2. 공통 계약 확정

반드시 맞춰야 하는 항목:

#### 데이터 계약
- 날짜 컬럼 기준
- 오일 종류 구분 기준
- split 방식
- 모델 입력 데이터 구조

#### 모델 계약
- 입력 시퀀스 길이
- 예측 horizon
- 추론 출력 형식
- 최대/최소 예측 유가 계산 가능 여부

#### 정책 계약
- threshold 기준
- recommendation 결과 단계
- 재학습 trigger 최소 범위

#### 백엔드 계약
- API 응답 필드
- recommendation 응답 구조
- 에러 응답 구조

#### 프론트엔드 계약
- 필요한 API 필드 목록
- 카드/차트 표시 기준
- 프론트는 계산하지 않고 표시만 수행한다는 원칙

완료 기준:
- 각 도메인이 입력/출력을 추측하지 않아야 한다.

## Codex 실행 방식

### 목표
통합된 프롬프트 세트를 바탕으로 Codex가 팀 리드처럼 작업을 진행하게 한다.

### 실행 원칙
- `AGENTS.md`를 최상위 운영 규칙으로 사용한다.
- 각 도메인 `init.md`를 역할별 프롬프트로 사용한다.
- `docs/05_shared/init.md`를 공통 계약 문서로 사용한다.
- 병렬 구현은 도메인 책임 범위를 넘지 않도록 수행한다.

### Codex에 반드시 전달되어야 하는 핵심 맥락
- 데이터 / 모델 / 백엔드 / 프론트엔드 분리
- LSTM 기반 모델
- 오일 종류와 무관한 공통 하이퍼파라미터 정책
- MLflow 사용
- threshold 기반 recommendation
- recommendation 로직은 백엔드
- 프론트엔드는 표시 중심
- data / model / backend는 `ener` 가상환경 사용

### 데이터

- 데이터 로딩
- 전처리
- 시계열 정렬
- split
- 모델 입력용 데이터셋 생성

---

### 모델
구현 범위:
- LSTM 모델 정의
- 학습 파이프라인
- 공통 하이퍼파라미터 반영
- MLflow 로깅
- 추론 출력 생성

---

### 백엔드
구현 범위:
- forecast API
- recommendation API
- monitoring API 골격
- recommendation 응답 구조
- 에러 응답 구조

---

### 프론트엔드
구현 범위:
- 대시보드 골격
- 예측 차트
- 최대/최소 유가 카드
- recommendation 카드
- monitoring 상태 표시 영역

---

## 병렬 작업 시 주의사항

- 데이터는 모델 입력 규격을 벗어나지 않는다.
- 모델은 백엔드가 필요로 하는 출력 필드를 누락하지 않는다.
- 백엔드는 recommendation 로직을 프론트엔드로 넘기지 않는다.
- 프론트엔드는 threshold 계산과 recommendation 판단을 직접 구현하지 않는다.
- 공통 정책은 여러 곳에 중복 하드코딩하지 않는다.
- data / model / backend는 동일한 `ener` 가상환경 기준으로 작업한다.

## 통합 검증 항목

병합 후 최소한 아래를 확인한다.

- [ ] 데이터 파이프라인이 모델 입력과 일치한다.
- [ ] 모델 출력이 백엔드 기대 형식과 일치한다.
- [ ] 백엔드 응답이 프론트엔드 기대 형식과 일치한다.
- [ ] recommendation 결과에 현재 시점, 최대 예측 유가, 최소 예측 유가가 포함된다.
- [ ] threshold 정책이 백엔드와 UI 설명에서 모순되지 않는다.
- [ ] Python 관련 구현이 모두 `ener` 가상환경 기준으로 동작한다.
