# Decision Log

## 문서 목적

이 문서는 프로젝트에서 내려진 중요한 의사결정을 기록한다.

이 문서의 목적은 다음과 같다.

- 왜 이런 구조와 정책을 선택했는지 남긴다.
- 나중에 같은 논의를 반복하는 비용을 줄인다.
- 구현과 문서의 방향성을 일관되게 유지한다.
- 여러 작업자가 동시에 작업할 때 기준점을 제공한다.

작은 구현 디테일보다는, 여러 도메인에 영향을 주는 결정 위주로 기록한다.

---

## 기록 규칙

각 의사결정은 가능하면 아래 구조를 따른다.

- 결정 ID
- 날짜
- 제목
- 상태
- 결정 내용
- 결정 이유
- 영향 범위
- 후속 조치

상태 예시:
- 초안
- 확정
- 대체됨
- 폐기

---

## 의사결정 목록

### D-001. 프로젝트는 문서 우선 방식으로 진행한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
구현보다 먼저 프로젝트 운영 문서와 도메인 문서를 정리하는 방식으로 진행한다.

#### 결정 이유
현재 프로젝트는 데이터, 모델, 백엔드, 프론트엔드가 동시에 얽혀 있다.
초기 문서 없이 구현부터 시작하면 도메인 간 계약이 쉽게 어긋날 수 있다.
Codex 단독 사용 환경에서는 문서가 실질적인 조율 장치 역할을 해야 한다.

#### 영향 범위
- 전체 프로젝트
- `AGENTS.md`
- `PLANS.md`
- `docs/` 전체 구조

#### 후속 조치
- 개요, 공통 기준, 도메인 문서를 우선 작성한다.
- 구현 시작 전 acceptance criteria를 먼저 정리한다.

---

### D-002. 프로젝트 책임은 데이터, 모델, 백엔드, 프론트엔드로 분리한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
프로젝트를 아래 4개 도메인으로 나누어 관리한다.

1. 데이터 모델링
2. 모델 학습
3. 백엔드 개발
4. 프론트엔드 개발

#### 결정 이유
각 영역의 변경이 서로 다른 산출물과 계약을 가지므로, 책임 경계를 먼저 세워야 병렬 작업이 가능하다.
도메인 분리가 약하면 threshold 정책, 추천 로직, 출력 포맷 등이 여러 곳에 중복될 수 있다.

#### 영향 범위
- 문서 구조
- 역할 분담
- 구현 디렉토리 구조
- 스킬 문서 구조

#### 후속 조치
- 도메인별 문서를 별도 디렉토리로 관리한다.
- API 계약과 추론 출력 계약을 명확히 분리한다.

### D-003. 예측 모델은 LSTM 기반으로 고정한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
유가 예측 모델은 LSTM 기반 모델을 사용한다.

#### 결정 이유
요구사항에서 명시적으로 LSTM 기반 모델 사용이 지정되어 있다.
따라서 초기 설계 단계에서 다른 모델 계열을 기본안으로 채택하지 않는다.

#### 영향 범위
- 모델링 문서
- 학습 코드 구조
- 실험 정책

#### 후속 조치
- `docs/02_modeling/lstm-modeling-spec.md` 작성
- LSTM 입력/출력 구조 정의

### D-004. 오일 종류와 무관하게 동일한 하이퍼파라미터 정책을 사용한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
오일 종류별로 अलग 하이퍼파라미터 세트를 두지 않고, 공통 하이퍼파라미터 정책을 사용한다.

#### 결정 이유
요구사항에 명시된 핵심 제약이다.
모델 관리 복잡도를 줄이고, 정책 일관성과 재현 가능성을 높일 수 있다.

#### 영향 범위
- 모델 학습
- MLflow 실험 관리
- 모델 비교 방식

#### 후속 조치
- `docs/02_modeling/hyperparameter-policy.md`에 명시
- 설정 파일 또는 문서에서 공통 정책 유지

### D-005. MLflow를 실험 추적의 기본 체계로 사용한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
모델 실험 추적과 관련된 기본 체계는 MLflow로 통일한다.

#### 결정 이유
요구사항에 MLflow 기반 모델 최적화와 성능 모니터링이 포함되어 있다.
실험 재현, 비교, 선택 근거를 남기기 위해 일관된 추적 체계가 필요하다.

#### 영향 범위
- 모델 학습
- 성능 모니터링
- 백엔드 연계 가능성

#### 후속 조치
- MLflow 관련 문서 작성
- 기록 항목과 선택 기준 정의

### D-006. 구매 추천 로직의 최종 판단은 백엔드에서 수행한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
threshold 기반 구매 추천의 최종 판단 로직은 백엔드가 수행한다.
프론트엔드는 계산된 결과를 표시하는 역할을 맡는다.

#### 결정 이유
프론트엔드에서 추천 로직을 재구현하면 정책 일관성이 깨지고 유지보수가 어려워진다.
추천 정책은 제품 로직으로 취급해야 하므로 백엔드에서 단일하게 관리하는 것이 적절하다.

#### 영향 범위
- 백엔드 API 계약
- 프론트엔드 UI 명세
- threshold 정책 문서

#### 후속 조치
- `docs/04_backend/recommendation-logic.md` 작성
- 추천 응답 스키마 정의
- 프론트엔드 문서에는 표시 책임만 남긴다.

### D-007. UI는 최대 예측 유가, 최소 예측 유가, 추천 결과를 반드시 보여준다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
사용자 화면은 최소한 아래 정보를 보여야 한다.

- 현재 판단 시점
- 미래 예측 구간 내 최대 예측 유가
- 미래 예측 구간 내 최소 예측 유가
- 구매 추천 결과

#### 결정 이유
요구사항에서 이 정보가 핵심 사용자 가치로 제시되어 있다.
모델 성능만 보여주는 것이 아니라 실제 구매 판단을 돕는 UI가 필요하다.

#### 영향 범위
- 모델 출력 정의
- 백엔드 응답 계약
- 프론트엔드 화면 설계

#### 후속 조치
- `docs/02_modeling/inference-output-spec.md` 작성
- `docs/04_backend/api-contract.md` 작성
- `docs/05_frontend/purchase-recommendation-ui-spec.md` 작성

### D-008. threshold는 문서화된 정책으로 관리한다.

- 날짜: 2026-03-25
- 상태: 확정

#### 결정 내용
구매 추천에 사용되는 threshold는 코드에 흩어진 매직 넘버로 두지 않고, 문서화된 정책으로 관리한다.

#### 결정 이유
추천 결과의 설명 가능성과 추적 가능성을 확보하기 위해서다.
threshold가 명시되어야 나중에 정책을 변경하거나 검증할 수 있다.

#### 영향 범위
- 모델 모니터링
- 추천 로직
- UI 설명 문구

#### 후속 조치
- `docs/03_mlflow_monitoring/threshold-rule.md` 작성
- 관련 코드에서 threshold 출처를 추적 가능하게 유지

### D-009. 최대/최소 예측 유가는 동일 forecast horizon 기준으로 계산한다.

- 날짜: 2026-03-25
- 상태: 초안

#### 결정 내용
현재 판단 시점에서 계산되는 최대 예측 유가와 최소 예측 유가는 동일한 forecast horizon 안에서 계산한다.

#### 결정 이유
UI와 백엔드 응답이 같은 기준 구간을 공유해야 사용자가 해석하기 쉽다.
기준 구간이 다르면 추천 근거가 흔들릴 수 있다.

#### 영향 범위
- 모델 추론 출력
- 백엔드 API 계약
- 프론트엔드 시각화

#### 후속 조치
- horizon 정의 후 확정 여부 재검토
- `docs/02_modeling/inference-output-spec.md`에 반영
